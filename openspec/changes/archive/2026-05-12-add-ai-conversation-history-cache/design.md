## 上下文
AI 对话当前入口页会先拉取 active 会话列表，并可能逐个关闭旧会话后再创建新会话。服务层 `createAIConversation()` 已包含关闭旧 active 会话的事务逻辑，但前端仍保留重复控制逻辑。详情页每次进入都会拉取最近消息，历史能力和首屏性能耦合在一起。

## 目标 / 非目标
- 目标：新进入 AI 对话默认快速创建新会话。
- 目标：旧 active 会话在新会话创建时可靠关闭。
- 目标：历史对话可查看，但不阻塞新对话首屏。
- 目标：历史列表和历史消息使用客户端缓存改善二次打开速度。
- 目标：历史 UI 在桌面和移动端均有明确的加载、空、错误与重试状态。
- 非目标：不改 AI provider、模型选择、质检导出或工单链路。
- 非目标：不引入新的服务端缓存系统。

## 决策
- 入口页 `/customer/conversations` 直接调用创建会话接口，不再先查询 active 会话。
- 新会话创建成功后跳转到详情页并追加 `?new=1` 查询参数；该参数只用于刚创建的会话一次性跳过首屏历史消息请求，不参与权限判断。详情页必须确认该 conversationId 与本次创建结果一致，使用后移除或忽略该标记。
- 关闭旧 active 会话以服务端事务为准；前端不再负责批量关闭旧会话。
- 旧 active 会话的关闭范围以认证用户 ID 为准，避免 `customerId` 与 `customerEmail` 混用导致关闭集合和列表集合不一致。
- 并发创建请求必须最终只留下一个 active 会话；实现可使用事务锁、重试或数据库约束，具体方式由实施阶段根据 Prisma/PostgreSQL 约束选择。
- 历史记录通过用户动作打开，首次打开时请求会话列表；缓存命中时先展示缓存数据，再后台刷新。
- 历史入口放在 AI 对话 header；桌面端使用侧边面板或弹层，移动端使用抽屉或全屏面板，不能遮挡输入框的基本发送能力。
- 历史列表必须支持 loading、empty、error、retry 状态。
- 历史列表首屏最多请求 20 条，支持“加载更多”；客户端列表缓存最多保留最近 50 条。
- 旧会话消息按 conversationId 缓存；打开旧会话时先显示缓存，再请求最新消息。
- 旧会话消息首屏最多请求最近 50 条，支持向上加载更早消息。
- 客户端缓存 key 必须包含认证用户 ID；消息缓存 key 必须同时包含认证用户 ID 和 conversationId。
- 退出登录或切换账号时，客户端必须清理或忽略上一个账号的历史列表和消息缓存。
- 新建、关闭、删除会话，以及新增消息或评分后，相关缓存必须更新或失效。
- 消息缓存最多保留最近 10 个会话，每个会话最多 100 条消息；超过上限时按最近使用时间淘汰。
- 持久化缓存序列化后总量不得超过 2 MB；写入接近或超过容量时，必须优先淘汰最久未使用的消息缓存，再淘汰历史列表缓存。
- 缓存写入遇到浏览器 quota 错误时，系统必须清理最久未使用缓存并继续保持核心对话功能可用。
- 缓存 TTL 为 24 小时；过期缓存不得作为可用数据展示，但可以被清理。
- 入口页到新会话可输入状态之前，客户端不得请求历史会话列表或历史消息。

## 风险 / 权衡
- 风险：跳过新会话初始消息加载后，如果 URL 误带新会话标记，可能短暂显示空内容。缓解：仅入口创建新会话后添加该标记，历史入口不添加。
- 风险：缓存可能短暂显示旧数据。缓解：缓存只作为首屏展示，后台刷新后替换。
- 风险：localStorage 或持久化 store 体积膨胀。缓解：限制历史列表、会话数、每会话消息数和 TTL。
- 风险：用户反复打开历史面板导致重复请求。缓解：同一用户同一资源的进行中请求应复用或忽略重复触发。
- 风险：用户手动保留或分享带 `?new=1` 的旧会话 URL。缓解：该标记只对刚创建的会话一次性有效，失配时按普通历史会话加载消息。
- 权衡：不引入 SWR 或服务端缓存，减少改动范围；后续可再统一缓存方案。

## 验证策略
- API/服务测试覆盖创建新会话会关闭旧 active 会话。
- API/服务测试覆盖并发创建后只保留一个 active 会话。
- API 测试覆盖未登录、跨用户历史列表和跨用户消息访问被拒绝。
- 页面或 hook 测试覆盖入口页不再拉取会话列表。
- 客户端状态测试覆盖历史列表缓存、消息缓存、账号切换隔离和 mutation 后刷新。
- 消息接口测试覆盖按需加载和分页参数不回退。
- UI 测试覆盖历史面板 loading、empty、error、retry 和移动端可用性。
- 性能测试或组件级网络断言覆盖新会话入口首屏不请求历史列表和历史消息。
- Store 测试覆盖缓存容量、TTL 和淘汰行为。
- Store 测试覆盖序列化字节上限和 quota 错误后的缓存淘汰。
- 页面测试覆盖 `?new=1` 只对刚创建会话一次生效，旧会话带该参数仍会加载消息。
