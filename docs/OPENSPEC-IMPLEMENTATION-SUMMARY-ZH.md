3. **创建变更提案**
   - 创建了 `add-self-hosted-faq-kb` 变更提案

   - 添加了对 OpenSpec 文档的引用

## 目录结构

```
openspec/
├── project.md              # 项目约定和上下文
├── specs/                  # 当前系统规范
│   └── faq-system/         # FAQ 系统规范
│       ├── spec.md         # 需求和场景
│       └── design.md       # 技术设计决策
├── changes/                # 提议的变更
│   └── add-self-hosted-faq-kb/  # 自建 FAQ 知识库实现
│       ├── proposal.md     # 原因、内容和影响
│       ├── tasks.md        # 实施清单
│       ├── design.md       # 技术决策（可选）
│       └── specs/          # 对现有规范的增量变更
│           └── faq-system/
│               └── spec.md # 更新后的需求
└── archive/                # 已完成的变更
```

## 使用 OpenSpec 的命令

```bash
# 列出所有规范
ls openspec/specs/

# 列出所有变更提案
ls openspec/changes/

# 查看特定规范
cat openspec/specs/[功能]/spec.md

# 查看变更提案
cat openspec/changes/[变更名称]/proposal.md
```

## OpenSpec 整体流程解释

OpenSpec 遵循三阶段工作流程：

### 第一阶段：创建变更
当需要添加新功能、进行架构变更或优化现有系统时，首先创建变更提案。提案包括：
- **proposal.md**：说明变更的原因、内容和影响
- **tasks.md**：详细的实施任务清单
- **specs/**：对现有系统规范的增量变更

### 第二阶段：实施变更
按照任务清单逐步实施变更，确保每个步骤都得到跟踪和完成。

### 第三阶段：归档变更
变更实施完成后，将其归档到 `archive/` 目录，作为项目历史记录的一部分。

## 结论

引入 OpenSpec 为该项目的需求管理提供了坚实的基础。它为如何定义和跟踪系统需求和变更带来了结构和清晰度，使得当前和未来的团队成员更容易理解系统并为其发展做出贡献。
